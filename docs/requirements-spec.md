# 要件定義書

## 1. 文書概要

### 1.1 目的

本書は、`harness-gacha` の MVP を実装するための要件を定義する。対象は、Claude Code / Codex 向けの harness pack をキュレーション型で販売するストアであり、TUI を中心とした発見体験、Web checkout による決済、購入後の導入体験、運営者による審査と掲載管理を含む。

### 1.2 対象範囲

本書が対象とするのは、MVP リリースに必要な機能要件、非機能要件、画面要件、運用要件である。pack 投稿の完全自動化、オープン marketplace、チーム課金、国際税務対応は対象外とする。

### 1.3 関連文書

- `docs/mvp-prd.md`
- `docs/ui-design.md`
- `docs/manifest-spec.md`
- `docs/review-checklist.md`

## 2. 背景

`skills`、`agents`、`hooks`、template などの harness asset は価値がある一方で、発見、比較、信頼判断、導入が難しい。MVP では、これらを `pack` として標準化し、購入前に安全性と互換性を確認できる TUI ストアを提供する。販売はランダム課金ではなく固定価格とし、ランダム要素は無料の discovery 体験に限定する。

## 3. システム概要

### 3.1 提供価値

- pack の発見体験を TUI で提供する
- pack の安全性、互換性、内容を購入前に判断できる
- Web checkout により購入できる
- 購入後に install 手順と配布物を取得できる
- クリエイターに売上分配できる

### 3.2 販売モデル

- 初期の販売主体はプラットフォーム運営者とする
- クリエイターは pack を提出する
- 運営者がレビュー後に掲載し、販売する
- 売上の一部を月次でクリエイターへ分配する

### 3.3 利用者区分

- 購入者
- クリエイター
- 運営者

## 4. スコープ

### 4.1 MVP に含む

- TUI カタログ
- 無料ランダム discovery
- pack 詳細表示
- pack 安全性詳細表示
- 購入案内画面
- Web checkout への遷移
- 購入済みライブラリ
- インストール詳細表示
- pack 審査と掲載管理
- 売上記録と月次分配計算

### 4.2 MVP に含まない

- 有料ガチャ
- コンプ要素
- creator 直接販売
- open self-serve marketplace
- ターミナル内カード入力
- チームプラン
- private registry
- 海外 creator 対応

## 5. 用語定義

- `pack`: 複数の harness asset をまとめた販売単位
- `manifest.json`: pack の内容、互換性、権限、導入方法を示す標準メタデータ
- `discovery`: 無料でランダムに 1 pack を表示する推薦機能
- `entitlement`: 購入者が pack を利用可能であることを示す権利情報
- `review`: pack を掲載前に確認する審査工程

## 6. 前提条件

- ローンチ対象は日本市場
- 決済は Web checkout を利用する
- pack は ZIP 形式で提供する
- 各 pack は `manifest.json` を含む
- クリエイターは招待制とする
- 運営者が手動レビューできる体制がある

## 7. 業務要件

### 7.1 購入者向け業務要件

- 購入者は pack を検索、閲覧、比較できること
- 購入者は無料でランダム discovery を実行できること
- 購入者は購入前に互換性、権限、外部通信の有無を確認できること
- 購入者は購入後に pack をダウンロードできること
- 購入者は購入後に install 手順を確認できること

### 7.2 クリエイター向け業務要件

- クリエイターは pack を提出できること
- クリエイターは売上分配対象として登録されること
- クリエイターは修正版 pack を再提出できること

### 7.3 運営者向け業務要件

- 運営者は pack を審査できること
- 運営者は pack を approve、reject、delist できること
- 運営者は売上と分配対象額を確認できること
- 運営者は返金時に entitlement を制御できること

## 8. 機能要件

### 8.1 カタログ機能

#### FR-001

- システムは featured、recent、recommended な pack 一覧を表示できること

#### FR-002

- システムはカテゴリまたはタグに基づいて pack を絞り込めること

#### FR-003

- システムは pack 一覧から pack 詳細へ遷移できること

### 8.2 Discovery 機能

#### FR-010

- システムは無料でランダムに 1 pack を表示できること

#### FR-011

- discovery 実行時に課金を発生させないこと

#### FR-012

- discovery 結果から pack 詳細へ遷移できること

#### FR-013

- discovery において当たり、ハズレ、排出率、コンプ要素を前提とした文言を使用しないこと

#### FR-014

- システムはガチャガチャを想起させる演出画面を持てること

#### FR-015

- discovery 演出はレバー操作、カプセル排出、結果カード表示のような段階表現を持てること

#### FR-016

- discovery 演出は短時間で完了し、ユーザー操作を過度に待たせないこと

### 8.3 pack 詳細機能

#### FR-020

- システムは pack 名、creator 名、価格、summary、対応ツール、対応バージョン、更新日を表示できること

#### FR-021

- システムは同梱物のカテゴリ別件数を表示できること

#### FR-022

- システムは必要権限と外部通信の有無を表示できること

#### FR-023

- システムは購入前に sample snippet または抜粋を表示できること

#### FR-024

- システムは購入前に prompt 全文、hook 全文、template 完全版、script 詳細実装を表示しないこと

### 8.4 安全性表示機能

#### FR-030

- システムは `shell`、`network`、`filesystem_read`、`filesystem_write`、`git` の権限有無を表示できること

#### FR-031

- システムは外部接続先の概要と利用上の注意を表示できること

### 8.5 購入導線機能

#### FR-040

- システムは Web checkout への遷移リンクを表示できること

#### FR-041

- システムは seller 情報、価格、返金ポリシー要約を購入前に表示できること

#### FR-042

- システムは TUI 内でカード情報入力を受け付けないこと

### 8.6 購入済みライブラリ機能

#### FR-050

- システムは購入済み pack 一覧を表示できること

#### FR-051

- システムは購入済み pack の version と更新有無を表示できること

#### FR-052

- システムは購入済み pack から install 詳細へ遷移できること

### 8.7 インストール案内機能

#### FR-060

- システムは install method、install command、手動導入手順を表示できること

#### FR-061

- システムは post-install 手順と注意事項を表示できること

#### FR-062

- システムは rollback または uninstall の注意事項を表示できること

### 8.8 審査機能

#### FR-070

- システムまたは運用は pack 提出物に対して review checklist を適用できること

#### FR-071

- 運営者は pack の review 結果を `pass`、`fail`、`needs follow-up` で記録できること

#### FR-072

- 運営者は pack を approve、reject、delist、suspend できること

### 8.9 売上分配機能

#### FR-080

- システムは pack ごとの gross sales、refunds、net basis を記録できること

#### FR-081

- システムは設定された creator share に基づき月次分配額を算出できること

## 9. 画面要件

### SR-001 ホーム / カタログ

- featured、recent、recommended を表示する
- 検索または絞り込み導線を持つ
- `ランダムで探す` 導線を持つ

### SR-002 Discovery

- ランダム推薦された 1 pack を表示する
- 詳細を見る、再実行、戻るの操作を持つ
- 待機、演出、結果表示の 3 状態を持つ
- レトロゲーム風のガチャガチャスクリーンとして成立する見た目を持つ

### SR-003 pack 詳細

- pack の判断材料を一覧表示する
- 購入前に全文表示しない
- 安全性詳細、購入案内への導線を持つ

### SR-004 安全性詳細

- 権限、影響範囲、外部通信の要約を表示する

### SR-005 購入案内

- Web checkout 導線と seller 情報を表示する

### SR-006 購入済みライブラリ

- 購入済み pack を一覧表示する

### SR-007 インストール詳細

- install command と導入手順を表示する

## 10. データ要件

### 10.1 pack データ

- pack は `manifest.json` を持つこと
- pack は一意な `id` を持つこと
- pack は version 管理されること

### 10.2 entitlement データ

- entitlement は user と pack の組み合わせで管理すること
- entitlement は購入、返金、無効化を識別できること

### 10.3 review データ

- review 結果、reviewer、review date、notes を保持できること

### 10.4 accounting データ

- 売上、返金、分配率、分配額を保持できること

## 11. 非機能要件

### 11.1 セキュリティ

- 購入前に危険権限を明示すること
- pack の権限情報は実挙動と矛盾しないこと
- credential exfiltration の疑いがある pack は掲載しないこと

### 11.2 可用性

- 少なくともカタログ閲覧、商品詳細閲覧、購入済み確認が安定して動作すること

### 11.3 性能

- カタログ画面は体感的に待ちが少ないこと
- pack 詳細表示は必要情報を即時に読めること

### 11.4 保守性

- pack 情報は `manifest.json` を基点に表示できること
- review 基準は文書化され、再利用できること

### 11.5 法務・表示

- 日本向け commerce disclosure を表示できること
- 有料ランダム販売と誤認される表現を避けること
- ブランド誤認を避けること

## 12. 運用要件

- 招待制 creator のみ受け付けること
- 公開前に手動 review を必須とすること
- delist や takedown の記録を残すこと
- 返金時の entitlement revocation 手順を用意すること
- 問い合わせ窓口を案内できること

## 13. 制約事項

- 決済は TUI 内で完結させない
- pack 詳細はコピー可能な粒度まで露出しない
- 初期リリースでは海外 creator を対象にしない
- open marketplace は採用しない

## 14. 受入条件

- TUI で pack を発見できること
- 購入前に安全性と互換性を判断できること
- 購入前情報だけで pack を再現できないこと
- Web checkout に正常遷移できること
- 決済完了後に entitlement が反映されること
- 購入済みライブラリから install 手順を確認できること
- 運営者が review と掲載可否判断を行えること
- 月次分配額を算出できること
